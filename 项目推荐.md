
# 校招生想走嵌入式 / Linux 底层，项目到底该怎么做？

这两个月陆陆续续看了不少校招生简历，越来越能看出一些非常明显的共性：
第一，很多人其实都是最近一年甚至半年内，才开始考虑转向嵌入式开发的“突击型选手”。
第二，项目经历也越来越模板化，往往就是一个 RK 平台上的应用开发项目，再加一个本科阶段做过的 MCU 项目。

看多了以后，你就会明白，很多简历不是项目少，而是项目几乎像从同一个模板里复制出来的。也难怪秋招面试官会越来越容易产生审美疲劳。

所以今天说说：如果你想往嵌入式、Linux 底层方向走，校招阶段到底该做什么项目，才更有辨识度，也更有面试价值？

首先，我推荐做 Linux 底软方向，原因并不复杂。AI 已经深度进入研发流程，代码生成、测试补全、文档整理、日志分析、问题排查都在被重塑。越是在这种时候，Linux 底层岗位的价值反而越清晰：系统是否可靠，不取决于代码能不能生成，而取决于你能不能真正理解硬件、内核、驱动框架和系统协同。AI 可以帮你补代码，但替代不了你去建立验证闭环、判断 patch 风险、处理复杂并发和系统级问题。所以，真正更抗替代的，不是“会不会写代码”，而是你有没有系统级工程判断力。

例如像下面这些方向：Linux 内核子系统开发、芯片厂 BSP 深度适配、量产级驱动维护和面向复杂业务场景的系统性能优化。它们并不是特别适合校招生直接硬碰硬的方向。因为这些工作本质上不是单点式编码，而是建立在真实硬件平台、完整软件栈、跨模块协同、长期工程迭代基础上的系统性研发活动。对于校招生来说，直接去碰这些“终局形态”的工作，难度过高，也不现实。

我更推荐这样一类项目：
特点：底层含量高、面试可以往深处追问、个人能做出来比较速成、离真实岗位又不太远。按这个标准，我把适合校招生做的项目分成三档。

## 第一档：最推荐不太依赖硬件，但非常能体现底层能力。
### 1）内核可观测性 / 诊断平台
不要只停留在“写一个 eBPF 脚本”这种层面。更像样的做法，是做一个围绕调度、内存、中断、网络延迟的分析工具链，把 tracepoint、ftrace、kprobe/kretprobe 串起来，再配一个用户态报告器或者分析前端。

这个方向的最大优点是：不依赖硬件、上手快、能快速出结果、还能一路往内核机制追问下去，如：你既能讲 tracing 框架本身，也能讲 hook 点怎么选、事件怎么组织、性能开销怎么控制、数据怎么从内核传到用户态。对校招生来说，这类项目的性价比非常高。

### 2）内核模块 + 用户态控制面项目
这一类我也很推荐。比如你可以做一个：内核资源监管器、异常行为检测器、内核态限流 / 统计模块；内核侧负责采集、限流、统计，用户态不要只用 ioctl 糊一个接口，尽量做成 Generic Netlink + CLI / 守护进程的形式。这种项目的价值，不只是“我写了个模块”，而是你能证明自己理解一整套：kernel、uapi、userspace 之间的交互边界，知道控制面该怎么设计、接口该怎么抽象、数据该怎么组织。这样一来，它就不是一个孤立的小实验，而更像一个初具工程形态的系统项目。

### 3）KUnit + fault injection 的“内核工程化项目”
这一类被很多人忽视，但我觉得其实值得做。你完全可以做一个“内核模块可靠性验证项目”：自己写模块、补 KUnit 测试、用 fault injection 人为制造失败路径去验证错误处理链路、资源释放逻辑和回滚流程。

大多数人做项目，只会讲 happy path：功能跑起来了，结果出来了，就结束了。但真正像系统工程师的人，会讲：失败路径、异常覆盖率、资源清理、恢复策略。你能把这些讲清楚，味道就完全不一样了。面试官会觉得你不只是“会写功能”，而是开始具备工程意识。

## 第二档：强推荐和真实嵌入式岗位贴得很近，面试价值也很高。
### 4）DMA / 零拷贝 / 高吞吐数据通路项目
如果你想让自己的项目更有“性能意识”，这一类非常合适。可以围绕一个采集链路或者传输链路去做：中断、DMA、环形缓冲、用户态读取；哪怕场景不大，只要路径完整，就已经很有价值。这一类项目很容易把问题引到更深的层面，比如：cache 一致性怎么处理；DMA 回调时序怎么保证；环形缓冲如何避免阻塞吞吐和时延之间怎么权衡；用户态读取路径有没有额外拷贝；瓶颈到底在中断、内存还是调度；也就是说，它天然适合被深入追问，面试收益很高。

这一档里，我还想额外强调一个判断
很多人简历里写的“驱动开发”，本质上其实只是 BSP 已支持前提下的板级适配：看原理图、改设备树、调通已有驱动；这当然不是没价值，但它更像适配能力，不是独立开发能力；更像会接线，不等于会造轮子。更有含金量的是官方 BSP 不支持的外设驱动开发。也就是从 0 开始去做：驱动编写、设备树补充、寄存器配置、初始化时序、数据收发、中断流程、错误处理、调试验证。这种项目会真正逼着你去看手册、啃时序、分析日志、定位问题，也更能体现底层能力。与其在一个项目里“移植一堆外设”，不如把一个外设真正做深、做透。十个浅层适配堆不出含金量，一个真正做透的驱动，反而更能让人看出你有没有底子。

基于这个标准，我更推荐优先考虑下面几类框架型方向。

### 5）V4L2 摄像头/视频链路项目
有些同学还在做“用 SoC 读一个温湿度传感器”这种项目。说实话，这类题目太轻了，很容易给人一种“大平台做小题目”的感觉：平台不低端，但项目本身没有把 Linux 框架能力体现出来。

如果要做驱动，真不如看看 V4L2 这种成熟框架。你不一定非要做完整产品，但哪怕只是做一个简化版的外设驱动 sensor、bridge、capture 链路，只要你能把这些关键链路讲明白：subdev、格式协商、buffer 流程、stream on/off 数据路径那就已经比“改设备树点亮一个外设”强很多了。

它的价值在于：你接触到的不是一个孤立的外设驱动，而是一整套媒体子系统的组织方式。面试官也很容易顺着这个项目往下追问，看你到底是真的理解框架，还是只是照着 BSP 改了点配置。

### 6）ASoC 音频项目
音频也是非常适合做“复杂一点的底层项目”的方向。因为 ASoC 天然就是框架型开发，不是那种孤零零的小模块。你会接触到这些层次：machine、platform、codec、DAI；也会碰到很多非常真实的系统问题：时钟配置、DMA 数据通路、音频路由、功耗管理、插拔检测

这个方向特别适合拿来证明：你不是只会照着模板把驱动跑起来，而是真正理解一个成熟子系统内部是怎么分层、怎么协作、怎么落到具体硬件上的。相比做一个简单传感器，这类项目更接近真实岗位中的 Linux 底层开发场景。

### 7）DRM / KMS 显示项目
显示方向属于更硬核的一类。它的优点是含金量高，缺点是门槛也明显更高。像下面这些概念：modeset、framebuffer、plane、CRTC、显示 pipeline、同步刷新；它们的理解和调试难度都明显高于一般外设驱动。因为它上手陡、链路长、调试复杂，如果基础还没打稳，很容易陷入细节泥潭；但如果你已经做过一个比较完整的驱动或子系统项目，再去补一个简化版显示链路，简历的质感会提升很多。

## 第三档：加分项不是主武器，但能拉开风格差异。
### 8）Rust for Linux 小型模块项目
这个方向现在最适合的玩法，不是“我要用 Rust 重写一切”，而是做一个很小但边界清晰的内核模块 / 驱动实验，把这些东西讲明白：安全抽象、绑定层、接口边界、模块组织方式

### 9）实验型操作系统项目 / OS 社区参与
这一类项目也可以作为加分项。比如接触或参与一些实验型或开源操作系统社区，如 RT-Thread、Tock、OpenHarmony、PolyOS、rCore-OS、Zephyr、NuttX 等。可以围绕引导加载、内存管理、中断处理、任务切换、系统调用、文件系统或者 RTOS 的移植、组件机制和社区协作流程展开学习与实践。

这类项目的价值，不在于你真的“做出了一个操作系统”，而在于它能体现你愿意继续往系统底层走，也确实接触过操作系统核心机制或开源 OS 生态。它更适合作为补充亮点。

这两类项目更适合作为补充亮点，而不太适合作为唯一主项目。它们更能体现你对技术趋势、系统底层和开源生态的关注，但不能替代一个真正能被深入追问的主项目。

---

## 最后总结
说到底，校招项目最怕的不是不够大，而是不够深；最怕的不是没做平台，而是没做出系统理解。认真做一个能被追问、能体现底层能力、能证明你有工程判断力的项目真的很重要。面试官最后真正想看的，也从来不是你“做没做过某个项目”，而是你有没有借这个项目，长出继续做 Linux 底层开发的兴趣和潜力。
