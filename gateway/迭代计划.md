
## 1. 设备抽象层 (Device Abstraction Layer)
- 定义统一的接口API（open/close/read/write/ioctl）
- 封装不同类型接口的特有参数设置
- 如串口的波特率设置、CAN的比特率、SPI的时钟频率等

## 2. 配置驱动架构
- 通过配置文件定义每个接口的参数
- 包括设备路径、接口类型、协议类型、通信参数等
- 启动时根据配置加载相应的处理模块

## 3. 插件化协议处理器
- 针对不同接口类型编写专门的驱动模块
- UART接口使用termios进行配置
- CAN接口使用SocketCAN API
- SPI/I2C接口使用标准的ioctl调用

## 4. 多线程/异步I/O模型
- 为每个接口创建独立的工作线程或事件循环
- 使用epoll/kqueue进行高效的I/O多路复用
- 避免单一接口阻塞影响其他接口

## 5. 数据缓冲与队列机制
- 为每个接口设置独立的输入输出缓冲区
- 使用环形缓冲区处理高速数据流
- 实现流量控制避免数据丢失

## 6. 协议栈分层处理
- 物理层：处理电气特性（波特率、电压等）
- 数据链路层：处理帧格式、校验等
- 网络层及以上：处理具体应用协议


抽象出多个层级，最底层是传输层，只关心字节流在设备上的收发，是对于不同接口的封装比如uart、can、spi、iic等，这一层内部对不同类型的接口进行封装，例如对于
UART：波特率、数据位、停止位、校验位、阻塞/非阻塞
CAN：比特率、过滤器、帧类型
SPI：mode、clock、bit order
I2C：速率、地址、时钟拉伸
等参数，然后提供不同的接口（例如读写、设置波特率、停止位等），让用户可以通过配置文件选择不同的配置，例如uart1的波特率为多少 can1的比特率是多少等。这一层只关系通信接口，不关心具体的协议内容，不关心帧头帧尾
命令内容 ACK 是否成功 初始化顺序。

然后到协议层，这一层负责设备协议的解帧组帧，比如这个蓝牙模块的帧头是什么，帧尾是什么，数据格式是什么、ack回复长什么样等等，这一层的处理逻辑是针对具体设备的协议进行解析和封装的。这一层通过特定规则进行协议解析合并然后由设备层调用。总结就是怎么把业务数据编码成协议帧
怎么从收到的数据里解出完整帧
怎么判断一帧是不是有效
怎么识别 ACK / NACK / 状态回复
占位符如何替换
长度字段如何计算，不关心字节流动方式，也不管业务流程，只负责私有协议的处理

最上一层则叫设备层，针对设备的生命周期和业务流程进行管理，负责初始化、配置、启动、停止、复位、状态检查、参数下发等，简而言之就是决定流程、步骤、状态切换，对设备进行控制，不直接关心字节细节，也不关心具体协议

同时配置文件也要分为多个，不能杂糅，传输配置文件专门定义传输层的相关参数，协议配置文件专门配置私有协议相关内容，支持多个不同的协议自定义添加，设备配置文件专门定义设备层的相关参数和流程，支持按照不同设备名字定义不同的设备配置，例如某个蓝牙模块配置某某波特率等  最后还要有一个整体的总配置文件，定义整个项目的大方向，例如使用mqtt还是dds?有哪些设备节点？分别是什么接口类型？mqtt的broker地址是什么？dds的话题名？数据库配置？等这些比较小而杂但是常改变的，就不用单独出配置文件，直接用总配置文件即可

层级关系
业务层 / 上层调用
        ↓
设备层（生命周期、初始化、配置、运行状态）
        ↓
协议层（组帧、拆帧、ACK、字段解析）
        ↓
传输层（UART/CAN/SPI/I2C 读写与参数）
        ↓
硬件
